# ユーザーストーリー（user_stories.md）

対象システム：**HOTEL PLANISPHERE** & **ローカルモック予約システム**

本ドキュメントは、ATDDスキル（Phase 4）に基づき、実装済み基本機能（US-01〜05）およびローカルで開発・検証された新規機能（US-06〜07）のユーザーストーリー定義です。

---

## 1. ユーザーストーリーと受け入れ基準

### 【US-01】出張ビジネスの通常予約（アクター: 健二）
* **ストーリー**: 
  出張者（健二）として、ホテルの予約ページから「出張ビジネスプラン」の予約詳細ページ（`reserve.html?plan-id=5`）を開き、平日1泊1名、かつ予算内であることを即時計算で確認した上で、不要な登録を行わずに「ゲスト予約」を完了させ、完了画面にユニークな予約番号が表示されること。
* **受け入れ基準**:
  1. ログインなしで `reserve.html?plan-id=5` にアクセスできる。
  2. 平日（火曜日等）を選択し、大人1名、1泊に設定した際、合計金額が「7,500円」と正確に表示される。
  3. お名前「健二」、連絡方法「メール」、アドレス「kenji-51@example.com」を入力し、予約を確定する。
  4. 確認画面（`confirm.html`）で情報が正しく表示され、「この内容で予約する」をクリックすると、完了モーダルが開き「ご来館、心よりお待ちしております。」と表示される。

### 【US-02】家族旅行での追加オプション乗算と土日加算（アクター: 佐々木）
* **ストーリー**:
  旅行者（佐々木）として、複数名・連泊の予約において「朝食バイキング（1人1泊+1,000円）」オプションを選択し、さらに土日を指定した際、部屋代のみに25%の土日加算が正確に上乗せされた総額が、確定前に画面上で確認できること。
* **受け入れ基準**:
  1. 「お得な特典付きプラン（基本7,000円）」で、大人2名、2泊、平日（火曜）を指定し、朝食オプションをONにした際、合計金額が `(7,000 * 2 * 2) + (1,000 * 2 * 2) = 32,000円` に即時再計算されること。
  2. 宿泊日を土曜日に変更した際、基本部屋代に25%加算され、合計が `Math.round(7,000 * 1.25) * 4 + 4,000 = 39,000円` に正確に再計算されること。

### 【US-03】会員ログインおよびプランの出し分け制御（アクター: 黒田）
* **ストーリー**:
  プレミアム会員（黒田）として、ログイン画面から正規情報で認証を行い、予約ページに遷移した際、一般には非表示の会員限定プラン（3プラン）を含む全10プランが表示され、予約フォームでページ遷移を行ってもログイン状態が保持され続けること。
* **受け入れ基準**:
  1. 会員ID・パスワードを入力してログインする。
  2. 予約プラン一覧画面で、プレミアム限定プラン（限定プランA〜C）が表示され、総プラン数が10件になっていることをアサートする。
  3. 任意のプランから予約ページへ遷移した際、ヘッダーにログイン中の会員名が表示されていること。

### 【US-04】会員限定プランの権限制御ゲート
* **ストーリー**:
  一般ユーザーまたは未ログインのゲストとして、直接プレミアム会員限定のプランURL（例：`plan-id=1`）にアクセスを試みた際、不正なアクセスとして処理され、ログイン画面へ強制リダイレクトされること。
* **受け入れ基準**:
  1. 未ログイン状態で `reserve.html?plan-id=1` を開こうとする。
  2. 画面が表示されず、強制的に `login.html` に遷移することを検証する。

### 【US-05】入力値バリデーションによる不備検知
* **ストーリー**:
  操作に不慣れなユーザー（翔太）として、名前を空欄のままにしたり、電話番号入力で「ハイフン入り（090-1234-5678）」の形式エラーの状態で送信を試みた際、送信がブロックされて予約画面に留まり、バリデーションエラー（必須抜け、パターン不一致）が検出されること。
* **受け入れ基準**:
  1. 氏名を空欄にして「予約を確認する」をクリックした際、ValidityStateの `valueMissing` が `true` になること。
  2. 電話番号にハイフンを入れて送信しようとした際、ValidityStateの `patternMismatch` が `true` になること。

### 【US-06】監査適合型領収書PDFとPIN付き照合システム（アクター: 健二・美咲）
* **ストーリー**:
  宿泊完了した出張者が、マイページまたは予約完了画面から領収書PDFをダウンロードした際、宿泊基本料金と食事代・税金の内訳が明記されており、さらに下部に「検証用予約ID」と「4桁の照合PIN」「照合用QRコード/URL」が印刷されていること。経理監査役（美咲）がQRコードをスキャンして照合PINを入力すると、ホテルの本物のデータ（宛名、支払額、宿泊完了/キャンセル済みのステータス）のみが画面に表示され、宿泊者の詳細な個人情報は非開示（ハッシュ化・非表示）のままで真正性が確認できること。
* **受け入れ基準**:
  1. 領収書に「基本部屋料金」「朝食代等の追加料金」「消費税率別の消費税額」の3つの内訳行が印字されていること。
  2. 領収書下部に「RSV-」で始まる一意のIDと、ランダムに割り振られた4桁のPINが印字されていること。
  3. 照合URLにアクセスすると、PINコード入力画面が表示され、3回誤入力するとアクセスが24時間ロックされること。
  4. 正しいPINを入力すると、宿泊ステータスと支払総額が表示され、宿泊者本人の氏名や同行者情報などの個人情報は表示されないこと（プライバシー保護）。

### 【US-07】インプットガードと文字補正（アクター: 翔太）
* **ストーリー**:
  誤入力をしやすいユーザー（翔太）が、予約フォームの電話番号やクレジットカード、郵便番号欄に全角数字やスペース、ハイフンを混ぜて入力したり、メールアドレスの前後に余分なスペースをコピペしてしまった際、システム側が送信ボタン押下時にこれらを自動クレンジングして正常値（半角数値・トリム後アドレス）に変換し、警告エラーを出さずにそのまま正常に予約を完了できること。また、氏名欄については、スペースを削除せず半角スペースに統一補正することで、チェックイン時の名義不一致を防止すること。
* **受け入れ基準**:
  1. 電話番号やカード番号欄に「０９０ー１２３４ー５６７８」や「090 1234 5678」と入力した際、送信時に自動的に「09012345678」へ補正されて送信されること。
  2. メールアドレス欄に「 shota@example.com 」とスペース交じりで入力した際、自動トリムされて「shota@example.com」として登録されること。
  3. 氏名欄に「山田　太郎」と全角スペースを入力した際、「山田 太郎」と半角スペースへ自動変換されること（削除されず、名義として成立していること）。
