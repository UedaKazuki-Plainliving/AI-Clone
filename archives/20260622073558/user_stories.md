# ユーザーストーリー定義 (user_stories.md)

対象システム：**ホテル予約システム（日付指定・空室プラン絞り込み機能）**

本ドキュメントは、実利用者クローン（上田様）および検証用複製クローン（田中さん、小林さん）との個別深層ヒアリングおよび専門家全体討議を経て合意された、品質向上に直結する10件 of 新規および既存仕様ギャップを埋めるためのユーザーストーリー定義です。

---

## 1. ビジネス出張者・現場運用者向けストーリー (US-01 〜 US-03)

### 【US-01】宿泊日指定による空室プランのダイレクト絞り込み (上田さん要望 - 当たり前品質)
* **役割**: 一般購入者 (上田様 / 出張者)
* **目的**: 宿泊日程が決まっているため、空室のない満室プランを最初から排除し、予約可能なプランだけを即座にダイレクトに一覧表示・比較できるようにして、不毛な詳細クリックを排除する。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: 指定した宿泊日に実際に予約可能なプランのみが表示されること
    Given ユーザーは宿泊プラン検索画面にアクセスしている
    When チェックイン日に「2026-07-10」を指定する
    And 「空室のあるプランを検索」ボタンを押す
    Then 一覧表示される宿泊プランの件数が「1件」となり、プラン名に「ビジネス出張応援プラン」が含まれること
    And 満室プラン（例: ファミリー観光プラン）は非表示または選択不可になっていること
  ```

### 【US-02】旅費規程上限（1万円）の事前ガードロック (上田さん要望 - 予防品質)
* **役割**: 一般購入者 (上田様 / 出張者)
* **目的**: 会社の旅費規程（宿泊上限10,000円）を超える高額プランの間違った予約を未然に防ぎ、経理担当からの叱責や差し戻し手戻りの恐怖をゼロにする。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: 旅費規程チェックボックスを有効化し、上限超えのプランの選択をガードする
    Given ユーザーはプラン一覧画面で「出張モード（旅費上限自動ロック）」を有効にしている
    When 日付「2026-07-10」で検索を実行する
    Then 宿泊料金が「10,001円」以上のプラン（例: スイートルーム特別プラン）の「このプランを選択する」ボタンが自動的に非活性化（disabled）されること
    And ボタンに「旅費規程上限オーバー」という警告バッジが表示されること
  ```

### 【US-03】会社宛電子領収書の事前指定とコーポレート決済自動連携 (上田さん要望 - 利用時品質)
* **役割**: 一般購入者 (上田様 / 出張者)
* **目的**: 領収書の宛名誤表記による手戻りを防ぐため、予約完了時にあらかじめ登録された会社名で電子領収書をPDF出力し、支払いデータをコーポレート決済へ直接自動連携する。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: 予約完了時に会社宛ての電子領収書PDFをプレビューし自動発行する
    Given ユーザーは「ビジネス出張応援プラン」の予約手続きを進めている
    When 支払オプションで「コーポレート決済」を選択する
    And 宛名欄に「プレインリビング株式会社」と入力して「予約を確定する」を押す
    Then 予約が正常に完了し、自動生成された領収書No「RC-101」が発行されること
    And 領収書のPDF宛名が「プレインリビング株式会社」、金額が「10,000円」として即座にダウンロード可能になること
  ```

## 2. PC出張者（田中さん）向けストーリー (US-04 〜 US-05)

### 【US-04】暗号化LocalStorageによる自動ドラフト保存と2時間自動消去 (田中さん要望 - 信頼性)
* **役割**: PC出張者 (田中さん / リテラシー低・ADSL回線)
* **目的**: 回線瞬断やPCフリーズ時に入力データが消失する不安を解消するため、入力中の予約内容をブラウザ側に自動で一時保存し、ブラウザを完全に閉じてから2時間経過後にセキュリティのため自動消去する。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: 入力途中でブラウザが強制終了した場合、再開時にドラフトデータから復元されること
    Given ユーザーは予約フォームで「宿泊者氏名（田中）」や「電話番号」を入力中である
    When フォーム入力開始から5秒が経過する（バックグラウンドで自動暗号化保存が実行）
    And ユーザーがブラウザタブを誤って閉じる
    And 2時間以内に再度予約フォームを開く
    Then 画面に「未完了の入力データがあります。復元しますか？」とダイアログが表示されること
    And 「はい」を選択した際、入力途中だった氏名および電話番号が自動でフォームに再入力されること
  ```

### 【US-05】共用PC検知時の自動保存機能強制ブロック (田中さん要望 - セキュリティ)
* **役割**: PC出張者 (田中さん / ロビーやカフェ等の共有端末利用)
* **目的**: 出張先のホテルロビーやネットカフェの共用PCから予約する際、一時保存された個人情報が他人に覗き見されるリスクを完全に排除する。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: 共有PCチェックボックスがオンの場合、LocalStorageへの書き込みを完全に無効化する
    Given ユーザーは予約画面で「共有パソコンを使用しています（自動保存オフ）」にチェックを入れる
    When 予約フォームに入力を開始する
    Then ブラウザの LocalStorage に暗号化されたドラフトデータが一切書き込まれていない（`localStorage.getItem` が常に `null`）であることを検証する
    And セッション終了（タブを閉じる）と同時に入力内容がメモリから完全消滅すること
  ```

## 3. シニア代理予約者（小林さん）向けストーリー (US-06 〜 US-10)

### 【US-06】5分有効期限SMS認証とWebOTPによる1タップ自動入力 (小林さん要望 - アクセシビリティ)
* **役割**: 高齢の代理予約者 (小林さん / スマホ・電波悪)
* **目的**: シニア層がスマホでSMSの4桁認証コードを確認して切り替える際の時間切れタイムアウトのパニックを解消するため、WebOTP APIを介してSMSのコードを1タップで自動入力できるようにする。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: ログイン時にSMS認証コードが送信され、WebOTP APIを介して1タップで自動入力されること
    Given ユーザーは「090-1234-5678」を入力し、認証コード送信ボタンを押している
    When スマートフォンに「認証コード: [4321]」を含むSMSが届く
    Then ブラウザ上に「メッセージから自動入力しますか？」というOSの確認ポップアップが出ること
    And ポップアップで「許可」を押した際、入力欄（`#otp-input`）に「4321」が即時に自動転記され、5分以内にログインが完了すること
  ```

### 【US-07】ブラウザ標準機能を利用したA4しおり印刷専用CSS（@media print）最適化 (小林さん要望)
* **役割**: 高齢の代理予約者 (小林さん)
* **目的**: 旅行当日のスマホ操作が不慣れな高齢の親のために、予約情報をA4用紙1枚の巨大フォント（30px以上）で綺麗に印刷した「手渡しのしおり」を作れるようにする。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: 予約完了画面を印刷した際、余計なヘッダーや広告が消え、しおりが極大フォントでA4横1枚に収まること
    Given ユーザーは予約完了画面で「旅のしおりを印刷する」を押す
    When ブラウザの印刷メディア（@media print）がエミュレートされる
    Then サイドナビゲーションやフッターが非表示（`display: none`）に制御されていること
    And ホテル住所、緊急連絡先、チェックイン時間が「font-size: 30px`」以上の極大フォントでA4用紙に綺麗にレイアウトされていること
  ```

### 【US-08】誤タップ防止のための2秒長押し確定（ホールド）決済ボタン (小林さん要望 - アクセシビリティ)
* **役割**: 高齢の代理予約者 (小林さん / 手の震え・誤操作あり)
* **目的**: 画面スクロール時や手の震えによる「意図しない予約確定（誤タップ）」での契約成立バグを防ぐため、2秒間ボタンをホールドし続けないと処理が実行されない安全ボタンを提供する。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: 確定ボタンを2秒間ホールドし続け、プログレスバーが100%に達した際のみ決済が実行されること
    Given ユーザーは支払金額内訳（基本料10,000円＋税1,000円）を確認している
    When 「長押し2秒で予約を確定する」ボタンを1回だけ短くタップする
    Then 決済処理は実行されず、エラーも発生しないこと
    When ボタンを「2,000ms」長押し（マウスダウン/タッチスタート）し続ける
    Then ボタンの `aria-valuenow` 属性値が「100」に達し、ボタンが `data-status="confirmed"` に遷移して決済処理が開始されること
  ```

### 【US-09】厨房アレルギー重要フラグ連携とハートビート同期監視 (小林さん要望 - 信頼性)
* **役割**: 高齢の代理予約者 (小林さん)
* **目的**: アレルギー（そば・甲殻類等）を持つ高齢の親の命を守るため、WEB予約時に入力した「食事制限要望」が、ホテルの調理場管理システムへ確実に整合性を持って同期され、漏れがないことを監視する。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: 予約時にアレルギー要望を入力し、厨房管理システムへハートビート監視下でデータ同期されること
    Given ユーザーは「食事アレルギー：そば」を選択して予約を確定する
    When 予約データが顧客DBに登録され、バックグラウンドの厨房システム同期APIが実行される
    Then 毎分実行される整合性監視サービス（ハートビート）により、顧客DBと厨房DBのアレルギーフラグ一致が検証されること
    And 同期エラーが発生した場合は、フロントスタッフ画面に赤色で「アレルギー未同期警告」が点滅表示されること
  ```

### 【US-10】4桁一時共有コードによるコールセンター簡易電話セッション引き継ぎ (小林さん要望)
* **役割**: 高齢の代理予約者 (小林さん)
* **目的**: WEB操作中に画面がフリーズしたりエラーが出た際、入力済みのホテル要望等を消さずに、電話でコールセンターに「4桁のコード」を伝えるだけで入力をスムーズに代行・引き継いでもらえるようにする。
* **受入基準 (Gherkin)**:
  ```gherkin
  Scenario: 入力途中画面で電話サポートをリクエストし、生成された4桁コードでスタッフに情報を引き継ぐ
    Given ユーザーは予約フォームの「禁煙室希望」「エレベーター近くの部屋希望」を入力完了している
    When 画面最下部の「お電話でのご予約引き継ぎ」ボタンを押す
    Then 画面に「あなたの引き継ぎコード: 【4321】」が巨大表示されること
    And オペレーターが管理画面で「4321」と入力した際、ユーザーがWEB上で入力途中だった「禁煙室希望」「エレベーター近」のJSONデータが安全にコールセンター側へ取得・ロードされること
  ```
