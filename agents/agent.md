# AIクローン共通設計書 (agent.md)

本ドキュメントは、AIクローン（エージェント）群の共通処理、プロンプト構造、エラーリカバリー方針、およびマルチエージェントオーケストレーション方式を整理したものです。

---

## 1. 共通プロンプト構造 (System Prompt Structure)

すべてのAIクローンは、以下の統一されたフォーマットに基づきシステムプロンプト（指示書）が設計されています。

```markdown
# 役割名
[クローンの名称とアイデンティティ]

# コンテキストと前提条件
- 対象システム: {system_name}
- 入力ソース: 実ユーザーアンケート / 既存の要件テキスト

# 主要任務
[クローンが果たすべき具体的なアウトプット定義]

# 制約事項
- 出力は必ず指定されたJSON構造またはフォーマットに従うこと
- 日本語で明確かつ論理的に表現すること
- 確証のない事実は「推定（確信度:低）」などのアノテーションを入れること
```

---

## 2. 共通APIインターフェースとデータモデル

バックエンド（FastAPI）と各エージェント間の通信は、スキーマの堅牢性を保証するために Pydantic モデルで共通化されています。

### 設定モデル (SettingsModel)
```python
class SettingsModel(BaseModel):
    provider: str        # 'mock', 'gemini', 'openai', 'anthropic'
    apiKey: str          # 暗号化/マスクされたAPIキー
    model: str           # 使用モデル名
    githubToken: str     # GitHub PAT
    githubRepo: str      # 'owner/repo' 形式
    githubBranch: str    # コミット先ブランチ
```

### シミュレーション出力モデル
シミュレータがエージェント群の議論を経て最終的に返却するJSONデータモデルです。

```json
{
  "domain": "string (システムドメイン分類)",
  "user_clones": "Array<UserClone>",
  "duplicate_clones": "Array<DuplicateClone>",
  "debate_logs": "Array<ChatMessage>",
  "atdd_gherkin": "string (Gherkinシナリオ)",
  "playwright_code": "string (テストコード)",
  "feasibility_feedback": "string (自動化検証結果)",
  "review_result": "ReviewResultObject",
  "director_summary": "string (指揮者総括)",
  "quality_metrics": "Map<string, number> (ISO/IEC 25010スコア)"
}
```

## 3. エラーハンドリング & フォールバックパターン (Error Recovery)

API接続切れ、タイムアウト、レートリミット（429）、またはJSONパースエラーに対処するため、共通エンジンに以下のロジックが組み込まれています。

```
[LLM API呼び出し]
       │
       ├── 成功 ───> JSONパース ───> 正常終了
       │               │
       │            パース失敗
       │               │
       │               ▼
       │         [正規表現抽出・クリーンアップ] ──> 成功 ──> 正常終了
       │               │
       │            抽出失敗 ──> [モックデータへフォールバック]
       │
       └── 失敗 (429/500/タイムアウト) ──> [モックデータへフォールバック]
```

1. **JSON抽出の堅牢化**: LLMの出力に markdown の ```json ... ``` などのノイズが含まれる場合、正規表現 `r"```json(.*?)```"` で抽出しトリミングします。
2. **モック・フォールバック**: APIエラーが発生した場合、利用者が指定したシステム名や要望に含まれるキーワード（「経費」「ホテル」「予約」「EC」など）を形態素解析的に評価し、該当するドメインの最適モックデータを瞬時に生成してシステムを無停止で維持します。

## 4. 実行フローとパイプライン (Orchestration Pipeline)

総合指揮者エージェントが、以下の順序で非同期的に処理をオーケストレーションします。

1. **分析フェーズ**: 入力されたアンケートから「利用者クローン」を生成。
2. **複製フェーズ**: 属性バリエーションを与えた「統計複製クローン」を作成し、一貫性とバイアスから信頼度を算出。
3. **対話フェーズ**: QAエキスパート（高木、高橋、上田）と利用者クローンの深い心理的葛藤対話ラリー（個別ヒアリングと討論）を経て抽出された要求・ユーザーストーリーを、プロジェクト責任者・品質責任者である「PM」が総合レビュー（品質寄与の有効性、インタビュー有効性、プロジェクト継続実現性を評価し、不適合時は差し戻し）して合意を形成します。
4. **エンジニアリングフェーズ**: 「自動化エンジニア（ATDDエンジニア）」が合意された要件から Gherkin シナリオと Playwright テストコードを生成し、5大技術観点（ロケーター、手順、期待結果、前提、データ）に基づいて自動化実装の実現性を検証・評価します。
5. **品質管理フェーズ**: 「品質管理者」が要件定義書の内容を ISO/IEC 25010 の品質特性（製品品質、利用時品質）にマッピングして詳細メトリクスを定量計測します。
6. **レポートフェーズ**: 指揮者が全体を総括し、妥当性判定報告書（Director Report）を出力します。
