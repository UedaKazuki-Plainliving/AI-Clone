# Skill: Quality Measurement

## 1. 目的
本スキルは、ATDD（受入テスト駆動開発）および要求探索プロセスによって構築されたシステムプロトタイプに対し、ISO/IEC 25010 規格に準拠した品質特性の測定および評価を論理的・定量的に実施するためのガイドラインである。
評価結果は、製品品質および利用時品質として測定方法、メトリクス定義、およびその具体的な算出根拠とともに開示されなければならない。

## 2. 測定対象の品質特性とメトリクス定義

### 2.1 製品品質 (Product Quality)
1. **機能適合性 (Functional Suitability)**:
   - **メトリクス**: 要求自動テストカバレッジ率（%） = (自動テストで合格した検証対象要求数) / (合意されたMVP要求総数) * 100
   - **測定方法**: テストランナーの実行結果と要求マトリクスの紐付けマッピングを検証。
   - **根拠**: 合意された全14要件（R-01〜R-14）に対応する Playwright E2E アサーションの実行ログ。
2. **信頼性 (Reliability)**:
   - **メトリクス**: E2Eテストパス率（%） = (正常パスしたテストケース数) / (総実行テストケース数) * 100
   - **測定方法**: 自動テストスイートの全テスト実行結果（Playwright runner stdout）を集計。
   - **根拠**: 既存テストスイートおよび新規追加テストスクリプトの実行ログ。
3. **使用性 (Usability)**:
   - **メトリクス**: 入力バリデーションエラー自動補正率（%） = (エラーとならずに自動補正または事前ガードされた入力項目数) / (検証対象の入力項目数) * 100
   - **測定方法**: 氏名スペース・全角数字等の補正ロジックの動作検証。

### 2.2 利用時品質 (Quality in Use)
1. **有効性 (Effectiveness)**:
   - **メトリクス**: タスク完了率 (Task Completion Rate - TCR)（%） = (タスクを正常に完了できたペルソナ数) / (検証に参加したペルソナ総数) * 100
   - **測定方法**: 各ペルソナ（健二、翔太、山本など）に定義された具体的な業務シナリオ（部屋食明細の経費精算、予約エラー時のリトライ等）をシステム上でシミュレーションし、達成判定を行う。
   - **根拠**: 翔太の領収書内訳精算が経理監査で承認されるか、山本の入力自動補正が働き離脱しないかのシミュレーションログ。
2. **効率性 (Efficiency)**:
   - **メトリクス**: 精神的オーバーヘッド削減時間（分/回） = (改善前のタスク所要時間) - (改善後のタスク所要時間)
   - **測定方法**: 各ペルソナのタスクごとの操作ステップ数、待ち時間、手計算による割り戻し時間の削減量を測定。
   - **根拠**: 健二が電卓で部屋食代を引き算する時間（5分）や、翔太が経理に再申請する手続き時間（30分）などの削減実績値。
3. **満足性 (Satisfaction)**:
   - **メトリクス**: SUS (System Usability Scale) スコア = 標準SUSアンケート（10問）による算出値（0〜100点）
   - **測定方法**: ワークショップに参加したユーザー・クローンの評価アンケートから算出。

## 3. 画面表示と成果物への統合
- 算出されたすべての品質特性は、測定方法と算出根拠とともに「品質メトリクス評価レポート（`quality_metrics.md`）」に出力されなければならない。
- ダッシュボードの「品質ダッシュボード」画面では、これらのメトリクスの測定方法と算出根拠を示す表や注記が、ユーザーにわかりやすく表示されなければならない。
