## 1. ユーザーストーリーマップ（行動 → 目的 → 不安 → 困りごと）

| 行動 (User Action) | 目的 (Goal) | 不安 (Anxiety) | 困りごと (Pain Point) |
| :--- | :--- | :--- | :--- |
| **プランの選定** | 条件・予算に合うプランの即時決定 | 選択したプランが実は満室で、予約処理の途中で弾かれるのではないか | 宿泊日を指定しても空室状況が一覧で分からず、プランを個別展開する必要がある |
| **予約情報の入力** | 誤りのない情報送信と一発完了 | 全角・スペースの誤入力でシステムエラーになり、入力がクリアされてやり直しになるのではないか | エラーメッセージが無慈悲で分かりにくく、コピペ入力したスペース等が原因だと気付けない |
| **領収書の発行・経費精算** | 経理監査の一発通過と二重請求リスクの排除 | 食事代込みの総額表示で経理に精算却下されるのではないか / PDFの偽造を疑われるのではないか | 領収書に消費税・オプション料金の内訳が明記されておらず、手計算で割り戻す必要がある。また、キャンセル後のPDFによる二重請求の監査ができない |

## 2. 成果物分類（要求工学モデル）

### 2.1 ユーザーストーリー & ネガティブストーリー
* **ユーザーストーリー**: 
  * [US-01〜05 (実装済み)](file:///C:/Users/plain/.gemini/antigravity/brain/752de375-5841-460e-a77e-ed98142b0ce3/user_stories_hotel.md#L9-L46) および [US-06〜07 (新規合意要望)](file:///C:/Users/plain/.gemini/antigravity/brain/752de375-5841-460e-a77e-ed98142b0ce3/user_stories_hotel.md#L48-L77) 参照。
* **ネガティブストーリー (不正・障害シナリオ)**:
  * **【NS-01】領収書の金額改ざん不正精算**: 悪意のある出張者が、ダウンロードした領収書PDFのHTMLソースを書き換えて金額を水増し請求する。
  * **【NS-02】キャンセル後の二重請求**: 予約をダウンロード直後にキャンセルし、キャンセル料が発生していないにもかかわらず、手元のPDF領収書で全額経費精算を通してしまう。
  * **【NS-03】ミドルネーム喪失によるチェックイン拒否**: 外国人出張者が「Michael Jordan」と入力した際、スペース除去のインプットガードが作動して「MichaelJordan」となり、パスポートと名義不一致でホテル側で本人確認できない。

### 2.2 暗黙要求 (Implicit Requirements)
* **税法準拠 (インボイス・消費税区分)**: 領収書は単なる総額ではなく、基本部屋代（10%または8%）、食事代（8%軽減税率等）、および宿泊税等の区分が分かれていること。
* **プライバシー保護**: オンライン照合時、URLから個人の宿泊日時や宿泊ルート（部屋タイプ）が他者に特定されないよう、PIN認証による防壁と表示情報の制限（名前・具体的な部屋情報の非表示）を行うこと。

### 2.3 リスク一覧 & エッジケース
* **【リスク1】ブルートフォース攻撃**: 領収書の照合PINコード（4桁）が単純なため、自動プログラムによる総当たりで他の予約情報が照合される。
  * *対策*: 同一セッションまたは同一予約IDで3回PINを間違えた場合、照合画面へのアクセスを24時間ロックする（R-19）。
* **【リスク2】テスト環境のAPI依存性**: 郵便番号からの住所自動入力を外部APIで行うと、テスト時のネットワーク状態によってテストがFlaky（不安定）になる。
  * *対策*: テストコード（Playwright）内では外部API通信をインターセプトしてモック化するか、必須テスト対象外にする。
* **【エッジケース】**: カレンダー入力において「過去の日付」や「3ヶ月以上先の日付」を指定した際に、JSでクライアント側制御するだけでなく、サーバー側でも防ぐ。

### 2.4 品質要求（非機能要件）
* **金額計算の正確性 (Functionality)**: 人数（1〜9）、泊数（1〜9）、曜日（土日25%加算）、オプション（1人1泊+1000円）が掛け合わされた金額の正確性（エラー許容度ゼロ）。
* **操作の容易性 (Usability)**: バリデーションエラーで画面を戻さず、フロントエンドで文字形式を自動補正すること。

## 3. ATDD化された受入条件（Gherkin形式）

### シナリオ1: ゲストによる出張ビジネスプラン予約（実装済み・自動テスト対象）
```gherkin
Feature: ゲスト予約機能と金額計算の検証

  Scenario: ゲストが出張ビジネスプランを通常予約する
    Given 未ログイン状態でホテルの予約ページ "reserve.html?plan-id=5" を開く
    When 宿泊日に「来週の火曜日」を指定する
    And 宿泊数を「1泊」に指定する
    And 人数を「1人」に指定する
    And 朝食バイキングオプションを「選択しない」にする
    Then 画面上の合計料金が「7,500円」であることを確認する
    When お名前に「健二」を入力する
    And 連絡方法に「メール」を選択する
    And メールアドレスに「kenji-51@example.com」を入力する
    And 「利用規約に同意して予約する」をクリックする
    Then 完了モーダルが表示され、「ご来館、心よりお待ちしております。」と表示されること
```

### シナリオ2: 家族予約における土日加算とオプション計算の検証（実装済み・自動テスト対象）
```gherkin
  Scenario: 家族でお得な特典付きプランを土曜から2泊、朝食付きで予約する
    Given 未ログイン状態でホテルの予約ページ "reserve.html?plan-id=3" を開く
    When 宿泊日に「来週の土曜日」を指定する
    And 宿泊数を「2泊」に指定する
    And 人数を「2人」に指定する
    And 朝食バイキングオプションを「選択する」にする
    Then 画面上の合計料金が「39,000円」であることを確認する
    # ※計算内訳: 基本7,000円 * 2人 * 2泊 * 1.25(土日加算) + 朝食1,000円 * 2人 * 2泊 = 35,000 + 4,000 = 39,000円
```

### シナリオ3: インプットガードによる自動補正（未実装・ATDDレビュー対象）
```gherkin
Feature: 入力値の自動補正（インプットガード）

  Scenario: 電話番号の全角およびハイフンが自動補正されること
    Given 予約ページを開いている
    When 電話番号欄に全角とハイフンを含む「０９０ー１２３４ー５６７８」を入力する
    And 予約送信ボタンを押す
    Then エラーにならず、自動的に「09012345678」に補正されて送信されること

  Scenario: 氏名の全角スペースが半角スペースに補正されること
    Given 予約ページを開いている
    When 氏名欄に「山田　太郎」と入力する
    And 予約送信ボタンを押す
    Then 自動的に「山田 太郎」に補正され、スペースが削除されないこと
```

### シナリオ4: オンライン照合機能付き領収書の発行（未実装・ATDDレビュー対象）
```gherkin

    Then 画面に「予約ステータス: 宿泊完了」「金額: 7,500円」が表示され、宿泊者氏名は表示されないこと
```
